# Cowrie SSH Honeypot with Wazuh SIEM Integration

**SSH honeypot (Cowrie) with Wazuh SIEM integration — custom decoders and detection rules for classifying attacker recon, brute-force, and command execution in real time.**

___

## Overview

This project simulates a small-scale detection pipeline similar to what a SOC analyst would operate in production:

- **Cowrie** impersonates a vulnerable Linux server (`ubuntu-web01`) on SSH port 2222, logging every login attempt and command an attacker runs.
- **Wazuh** ingests those logs, parses them with a custom decoder, and applies custom rules to classify events by severity — from simple recon to brute-force detection to command execution.

Built as two isolated VirtualBox VMs (honeypot and SIEM manager kept separate, mirroring real-world log-integrity practice — you don't want your monitoring system living on the box that might get compromised).

___

## Architecture

```
                     ┌─────────────────────────┐
   Attacker SSH      │   Cowrie Honeypot VM    │
   traffic           │   (Kali Linux)          │
   ───────────────►  │   - Cowrie on :2222     │
                     │   - Fake hostname:      │
                     │     ubuntu-web01        │
                     │   - Logs to cowrie.json │
                     └───────────┬─────────────┘
                                 │ Wazuh Agent
                                 │ forwards logs
                                 ▼
                     ┌─────────────────────────┐
                     │   Wazuh Manager VM      │
                     │   - Wazuh Indexer       │
                     │   - Wazuh Manager       │
                     │     (custom decoder +   │
                     │      custom rules)      │
                     │   - Wazuh Dashboard     │
                     └───────────┬─────────────┘
                                 │ HTTPS :443
                                 ▼
                     Analyst views alerts in
                     Wazuh Dashboard (Threat Hunting)
```

Both VMs run on a shared VirtualBox **NAT Network**, isolated from the host's home network, with port-forwarding rules exposing SSH admin access, the honeypot's SSH port, and the dashboard to the host machine only.

## Tech Stack

- **Honeypot:** [Cowrie](https://github.com/cowrie/cowrie) (SSH/Telnet)
- **SIEM:** [Wazuh 4.9.2](https://wazuh.com/) (Manager, Indexer, Dashboard)
- **Virtualization:** VirtualBox, dual Kali Linux VMs
- **Networking:** VirtualBox NAT Network + port forwarding

## Detection Rules Built

Custom rules (`local_rules.xml`) classify Cowrie's JSON events by severity:

| Rule ID | Level | Trigger | Description |
|---|---|---|---|
| 100010 | 3 | Any `cowrie.*` event | Baseline honeypot activity detected |
| 100011 | 5 | `cowrie.login.failed` | Failed login attempt |
| 100012 | 10 | 5x rule 100011 in 60s | Correlated brute-force attack detected |
| 100013 | 7 | `cowrie.login.success` | Attacker successfully "logged in" |
| 100014 | 12 | `cowrie.command.input` | Attacker executed a command inside the fake shell |

A custom decoder (`local_decoder.xml`) parses Cowrie's raw JSON log lines using Wazuh's `JSON_Decoder` plugin so these fields become queryable in the dashboard.

| Successful login with guessed credentials | T1078 – Valid Accounts |
| Command execution post-login (`whoami`, `uname -a`, etc.) | T1059 – Command and Scripting Interpreter |
| Attempted file download (`wget`) | T1105 – Ingress Tool Transfer |

## Notable Challenges & Fixes

Worth highlighting — these were the genuinely non-obvious problems solved during the build:

- **Cowrie's file layout changed** in newer versions — `cowrie.cfg.dist` moved to `src/cowrie/data/etc/`, and the old `bin/cowrie` startup script no longer exists; `pip install -e .` is required to register the CLI entry point.
- **VirtualBox default NAT mode isolates each VM** — switching to a shared **NAT Network** was required for the honeypot and Wazuh manager to communicate, plus port-forwarding rules to reach both from the host.
- **Wazuh's JSON decoder always reports as `json`**, not a custom decoder name — rules must match on the actual event field content (`eventid`) rather than `decoded_as`.
- **`same_source_ip` only works with the reserved `srcip` field** — since Cowrie's JSON uses `src_ip`, correlation rules had to rely on the base `frequency`/`if_matched_sid` mechanism directly.
- **Disk space exhaustion** — Wazuh's Vulnerability Detection module silently downloaded ~18GB of CVE feed data, filling the VM's disk and causing the API service to fail; resolved by clearing the cache and disabling vulnerability-detection for this lab's scope.

## Setup Summary

1. Two VirtualBox VMs (Kali Linux) on a shared NAT Network.
2. Cowrie installed under a dedicated unprivileged `cowrie` user, configured with a believable hostname, listening on port 2222.
3. Wazuh manager, indexer, and dashboard installed via the official all-in-one installer on the second VM.
4. Wazuh agent installed on the Cowrie VM, configured to monitor `cowrie.json`.
5. Custom decoder and rules deployed on the manager to parse and classify Cowrie events.
6. Verified end-to-end with live SSH sessions against the honeypot.
